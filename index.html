1. 프로젝트 소개

1.1 개발 배경

성인 ADHD는 단순한 집중력 부족이 아닌 신경생물학적 어려움의 복합체입니다. 과제 시작의 어려움(해야 할 일 앞에서 멈춤), 동기 유지 불가(왜 해야 하는지 잊음), 시간 감각 왜곡, 작업 기억 결함(약속·대화 내용 유실), 막연함으로 인한 회피(큰 목표 앞에서 압도) 등 기존 할일 관리 앱이 전혀 다루지 않는 문제들이 일상을 지배합니다.

일상투두는 이 각각의 어려움에 1:1로 대응하는 기능을 설계했습니다. 과제 시작이 어려우면 집중 실행이 "지금 할 수 있는 가장 작은 것 하나"를 추천하고, 동기가 사라지면 원동력 새기기로 "왜 해야 하는지"를 상기시키며, 작업 기억이 부족하면 관계 기록이 대화와 약속을 즉시 저장하고, 막연한 목표 앞에서 회피하면 AI 플래닝이 실행 가능한 단위로 분해합니다.

설계 전반에는 "비난 없는 동반자"라는 톤을 일관되게 적용했습니다. 낮은 진입장벽(시작에 에너지가 거의 들지 않을 것), 즉각적 피드백(작은 행동에도 긍정 강화), 비판단적 환경(미완료에 죄책감 대신 완료에 격려)이라는 세 가지 원칙으로, ADHD 성향 사용자가 자기효능감을 회복하며 일상을 관리할 수 있도록 돕습니다.

1.2 핵심 기능

앱은 크게 3개 도메인으로 구성되어 있으며, 각 도메인 안에 전용 화면들이 배치됩니다. 목차(Table of Contents) 기반 네비게이션으로 원하는 기능에 빠르게 접근할 수 있습니다.

🗓️ 계획 세우기 (Planning)

  • 하루 계획하기 (Daily Planner): 오늘의 할일을 오전/오후/저녁 시간대별로 배치, 아이젠하워 매트릭스 기반 우선순위 분류, 보상·칭찬 시스템
  • 일정 계획하기 (Timeline): 시간축 기반 일과 관리, 미완료 일정 자동 추적, 빈 시간대 자동 감지("이 시간에 뭐 했어요?")
  • 내 계획 보기 (AI Plan): 프로젝트 기반 할일 관리, 진행률 시각화, 상태별 필터링 (전체/시작안함/진행중/중단/완료)
  • AI로 계획하기 (AI Chat): 4개 LLM 프로바이더 기반 ADHD 친화적 CAPS 메서드 (Chunk-Anchor-Pressure-Start Ritual)
  • Claude 연결하기: Claude Desktop MCP 연동으로 AI 할일 자동 생성
  • 집중 실행하기 (Execute): 타이머 + 원동력 상기 → 과제 시작의 마찰을 줄여 첫 발을 내딛도록 지원
  • 마음 깨우기 (Banner): 동기 부여 문구 + 미완료 할일 알림으로 실행력 자극

💡 생각과 기억 (Thoughts & Memory)

  • 원동력 새기기: 개인 맞춤형 동기 부여 문구 관리, 기쁨/감사/각성/결단 4가지 유형, XP·레벨 시스템, 할일 전환 기능
  • 관계 기록하기: 소중한 사람과의 만남·대화 기록, 관계/역할/부서 필터링, 연락 우선순위 자동 추천
  • 소식 챙기기 (PRO): 기록된 소식 시간순 조회, 연락 빈도 시각화, 소홀한 관계 파악
  • 연락 돌아보기 (PRO): 연락 빈도와 관계 인사이트 분석

💚 일상 돌보기 (Daily Care)

  • 감사 기록하기 (PRO): 사람별 감사 저널링, 긍정 경험 의도적 회상으로 정서 균형 회복
  • 활동 살펴보기 (PRO): 활동 패턴과 생산성 분석, 시간 사용 인사이트

🤖 AI & 결제 시스템

  • AI Gateway: Claude Sonnet 4, GPT-4o, Gemini 2.0 Flash, Groq — 4개 LLM 멀티 프로바이더 통합
  • MCP 서버: Model Context Protocol 구현, ChatGPT Actions / Claude Desktop 연동
  • 구독 결제: Paddle(웹) + RevenueCat(모바일) 크로스플랫폼 구독 관리

3. 아키텍처 및 기술적 특징

3.1 모노레포 멀티플랫폼 아키텍처

Turborepo + npm workspaces 기반 모노레포로 Web, Desktop, Mobile 3개 플랫폼을 관리합니다. 웹과 데스크탑은 하나의 Next.js 코드베이스를 공유하고, 모바일은 React Native로 네이티브 성능을 제공합니다.

  • apps/web/ — Next.js 웹앱 (Vercel 배포 + Electron 빌드 공유)
  • apps/desktop/ — Electron 메인 프로세스 (TypeScript)
  • apps/mobile-rn/ — React Native 앱 (iOS/Android)
  • BUILD_TARGET 환경변수로 웹/Electron 빌드 분기 처리

3.2 상태 관리 전략

Zustand를 중심으로 17개 스토어(8,000줄+)를 운영하며, 플랫폼별로 최적화된 영속화 전략을 적용합니다.

  • 웹: Zustand + Supabase Realtime (멀티 디바이스 실시간 동기화)
  • 데스크탑: Zustand + electron-store (세션 영속화, JWT 백업)
  • 모바일: Zustand + MMKV persist (네이티브 고성능 스토리지)

주요 스토어: todoStore, projectStore, authStore, pomodoroStore, aiPlanningStore, cherishedPeopleStore, relationshipStore, subscriptionStore, settingsStore, adhdStore, noteStore, dailyReflectionStore 등

  • 애니메이션: Framer Motion(웹), Reanimated 4 + Skia(모바일)로 플랫폼별 최적화
  • 드래그 앤 드롭: @dnd-kit으로 할일 순서 변경, 접근성(키보드) 지원
  • 다크 모드: next-themes(웹), NativeWind(모바일)로 시스템 설정 연동
  • 마케팅 랜딩페이지: 스크롤 기반 배경색 전환 (useScroll/useTransform), Smart CTA

3.4 Electron 데스크탑 아키텍처

Next.js 정적 빌드를 Electron에서 로드하는 구조로, 웹과 코드베이스를 공유하면서 데스크탑 네이티브 경험을 제공합니다.

  • Custom Protocol: app:// → apps/web/out/ 디렉터리 매핑으로 정적 파일 로딩
  • OAuth: 로컬 HTTP 서버를 띄워 OAuth 콜백을 받고, IPC로 메인 프로세스에 전달하는 PKCE 패턴
  • 세션 관리: electron-store로 JWT 토큰 영속화, Supabase 세션 백업
  • 빌드 분기: BUILD_TARGET=electron → output: 'export' 정적 빌드 활성화

3.5 결제 아키텍처

웹과 모바일에서 동일한 구독 상태를 유지하기 위해 Paddle(웹)과 RevenueCat(모바일)를 통합했습니다.

  • Paddle Webhook: HMAC-SHA256 서명 검증, 구독 라이프사이클 관리 (553줄 Edge Function)
  • 크로스플랫폼 동기화: Paddle → RevenueCat API 연동으로 웹↔모바일 구독 상태 통합
  • 구독 이력: subscriptions, users, subscription_history 다중 테이블 동기화

3.6 AI 아키텍처

멀티 프로바이더 게이트웨이로 4개 LLM을 추상화하고, MCP 서버로 외부 AI 도구와 연동합니다.

  • AI Gateway: 프로바이더 팩토리 패턴으로 Claude/OpenAI/Gemini/Groq 통합 (850줄+)
  • Tool Use: 스트리밍/비스트리밍 지원, ADHD 플래닝 특화 시스템 프롬프트
  • MCP Server: JSON-RPC 2.0 기반 MCP 2024-11-05 스펙 구현 (1,300줄+)
  • ChatGPT 연동: OAuth 기반 ChatGPT Actions로 일상투두 데이터 직접 관리

3.7 트러블슈팅 & 문제 해결

모노레포 멀티플랫폼 아키텍처와 결제 시스템 구축 과정에서 만난 주요 기술적 문제와 해결 방안입니다.

1) Electron OAuth 인증 (로컬 HTTP 서버 + PKCE)
  • 문제: Electron BrowserWindow에서 Google/Kakao OAuth 콜백 URL이 앱으로 돌아오지 않음
  • 원인: 데스크탑 앱은 웹 리다이렉트 방식의 OAuth 콜백을 직접 수신할 수 없음
  • 해결: 임시 로컬 HTTP 서버를 띄워 콜백을 받고, IPC 통신으로 메인 프로세스에 세션 전달. electron-store에 JWT 영속화
2) React Native 환경 설정 (Reanimated + NativeWind + MMKV)
  • 문제: RN 0.83 + Reanimated 4 + NativeWind 4 + MMKV 4 조합에서 빌드 실패 및 런타임 크래시
  • 원인: Reanimated 4의 worklets 플러그인 순서 충돌, MMKV v4 peer dependency 누락, Metro 경로 별칭 미작동
  • 해결: react-native-worklets/plugin을 babel 배열 마지막에 배치, react-native-nitro-modules peer dep 추가, babel-plugin-module-resolver로 @/ 경로 별칭 구현
3) Paddle↔RevenueCat 크로스플랫폼 구독 동기화
  • 문제: 웹(Paddle)에서 결제한 사용자가 모바일에서 구독 상태 미인식
  • 원인: 두 결제 시스템이 독립적으로 동작, 구독 상태 공유 불가
  • 해결: Paddle Webhook → Supabase DB → RevenueCat API 연동 파이프라인 구축 (553줄 Edge Function)
4) 모노레포 빌드 분리 (BUILD_TARGET)
  • 문제: Web/Electron 빌드 시 Next.js SSR, export 모드 충돌 + 모노레포 의존성 관리
  • 원인: Next.js output 설정이 단일 값만 지원, Turborepo 태스크 간 빌드 캐시 오염
  • 해결: 환경변수 기반 동적 next.config.ts, Turborepo 태스크별 출력 디렉터리 분리, cross-env 빌드 스크립트
5) Optimistic Update + Realtime 충돌
  • 문제: 드래그앤드롭 후 Realtime 이벤트가 상태를 덮어쓰며 UI 깜빡임 발생
  • 원인: 낙관적 업데이트와 서버 Realtime 응답의 타이밍 불일치
  • 해결: 로컬 mutation ID 추적, 자신이 발생시킨 이벤트 필터링하여 중복 적용 방지
